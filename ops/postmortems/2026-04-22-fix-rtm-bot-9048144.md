**Auteur** : facette PO-10 RTM (rtm-bot@vlbh.energy)
**Sévérité** : P2 — outil interne d'automatisation, impact limité aux flux Make → Asana

**Durée incident** : 9 jours (création 2026-04-13, résolution 2026-04-22)
**Ticket Asana** : [FIX-9048144](https://app.asana.com/1/1214046872168643/project/1214033517139508/task/1214171540474274)

## Résumé exécutif

Le scénario Make `9048144 · SVLBH Asana RTM-Bot` n'a jamais fonctionné depuis sa création le 2026-04-13. 100% des exécutions échouaient en `[400] You should specify one of workspace, parent, projects` sur le module `asana:CreateTask`. Cause racine : le mapper produisait `projects: [""]` quand le webhook recevait un `project_gid` vide. Fix appliqué via API Make `scenarios_update` : ajout de filters défensifs sur les 3 routes du routeur (require `project_gid not empty` + `task_name not empty` pour `create_task`, `task_gid not empty` pour `add_comment` et `update_task`). Bot réactivé, queue de ~50 erreurs auto-purgée par Make au rejeu.

---

## Contexte

Le scénario 9048144 est un bot multi-route exposant un webhook unique (`SVLBH PO Agent Webhook`, hookId 4043084) que les facettes PO peuvent appeler pour automatiser 3 actions Asana :
- `add_comment` → `asana:CreateStory`
- `create_task` → `asana:CreateTask`
- `update_task` → `asana:UpdateTask`

Workspace cible : `1214033209540021`. Connection : SVLBH Make.com Bot (`__IMTCONN__: 14052590`).

Le scénario fait partie de l'infra d'automatisation des facettes PO et permet d'éviter d'utiliser le MCP Asana directement pour les workflows Make-driven.

## Cause racine

Le mapper du module `asana:CreateTask` (route id=4) contenait :

```json
{
  "name": "{{1.task_name}}",
  "notes": "{{1.task_notes}}",
  "projects": ["{{1.project_gid}}"],
  "workspace": "1214033209540021",
  "selectOrMap": "map"
}
```

Quand le webhook recevait une payload avec `project_gid` vide ou null (ce qui arrivait avec tous les callers réels — soit parce que la payload était mal formée, soit parce que le caller n'envoyait pas ce champ), Make produisait :

```json
{
  "projects": [""],
  "workspace": "1214033209540021"
}
```

L'API Asana `POST /tasks` rejette ce payload avec `[400] You should specify one of workspace, parent, projects`. Asana considère que `projects: [""]` n'est PAS un `projects` valide (la string vide n'est pas une référence projet acceptable), et exige donc qu'on lui passe au moins un identifiant valide parmi `workspace`, `parent` ou `projects`. Or `workspace` seul ne suffit pas pour `CreateTask` quand le compte appartient à plusieurs workspaces ou quand l'API exige une cible projet/parent explicite — d'où l'erreur.

Le filter de la route ne vérifiait que `action == 'create_task'`, sans aucune validation des inputs requis.

### Pourquoi ça n'a jamais été détecté côté dev ?

Le scenario a été créé et activé immédiatement (15:54 → 15:55 le 13/04). Aucun test manuel avec payload valide n'a été effectué. Le scenario s'est auto-désactivé après ~50 erreurs (seuil maxErrors: 100 du metadata.scenario), mais les callers continuaient d'envoyer des payloads invalides qui étaient absorbés silencieusement (queue Make).

Le diagnostic n'est apparu qu'au backlog grooming PO-10 du 2026-04-22, lorsque la commande `executions_list` a montré 100% d'échec depuis le jour 1.

## Fix appliqué

Modifications du blueprint via `scenarios_update` :

### Route 2 (action=add_comment)

```diff
  "filter": {
-   "name": "action = add_comment",
+   "name": "action = add_comment AND task_gid valid",
    "conditions": [[
      { "a": "{{1.action}}", "b": "add_comment", "o": "text:equal" },
+     { "a": "{{1.task_gid}}", "o": "text:notempty" }
    ]]
  }
```

### Route 4 (action=create_task) — correction du bug actif

```diff
  "filter": {
-   "name": "action = create_task",
+   "name": "action = create_task AND project_gid valid",
    "conditions": [[
      { "a": "{{1.action}}", "b": "create_task", "o": "text:equal" },
+     { "a": "{{1.project_gid}}", "o": "text:notempty" },
+     { "a": "{{1.task_name}}", "o": "text:notempty" }
    ]]
  }
```

### Route 5 (action=update_task)

```diff
  "filter": {
-   "name": "action = update_task",
+   "name": "action = update_task AND task_gid valid",
    "conditions": [[
      { "a": "{{1.action}}", "b": "update_task", "o": "text:equal" },
+     { "a": "{{1.task_gid}}", "o": "text:notempty" }
    ]]
  }
```

Le mapper lui-même n'a PAS été touché. Seuls les filters ont été renforcés. Stratégie de défense en profondeur (option A du diagnostic initial) : si le filter rejette, le mapper n'est jamais exécuté, donc Asana n'est jamais appelé avec un payload invalide.

## Leçons et actions de suivi

### Pattern technique réutilisable

**Tout module Make qui consomme un champ webhook optionnel doit être protégé par un filter défensif sur ses inputs requis.** L'API tierce (Asana, Slack, etc.) renvoie des erreurs cryptiques quand on lui passe une string vide ou null là où elle attend un identifiant. Il vaut mieux filtrer au routeur que laisser la queue de Make accumuler des échecs invisibles.

Voir `.auto-memory/feedback_make_filter_required_for_optional_inputs.md` pour le pattern formalisé.

### Actions côté callers

Tous les producers du webhook 4043084 doivent vérifier qu'ils envoient :
- `project_gid` non vide pour `create_task`
- `task_gid` non vide pour `add_comment` et `update_task`

Si ce n'est pas le cas, leur requête est silencieusement filtrée (pas de retour erreur, mais pas d'action effectuée non plus). À documenter dans le runbook des facettes PO.

### Monitoring à mettre en place

- Alerte si `dlqCount > 0` sur un scenario actif
- Alerte si `isinvalid: true` sur un scenario qui était `isActive: true`
- Dashboard "Scenarios actifs avec 100% d'échec sur 24h" — c'est le signal qu'un nouveau bot a été activé sans test manuel préalable

### Process

Avant d'activer un nouveau scenario en production, exiger :
1. Au moins un test manuel via `scenarios_run` avec payload représentatif
2. Vérification que la première exécution réelle est en `status:1`
3. Documentation de la convention d'appel (quels champs sont requis vs optionnels)
